## 목표와 기준선

현재 dev는 TP 376, FP 2, FN 124로 precision 99.47%, recall 75.20%다. recall 80%에는
500개 positive 중 최소 400개를 찾아야 하므로 순증 TP 24개가 필요하다.

| 지표 | 현재 | 완료 게이트 |
| --- | ---: | ---: |
| dev recall | 75.20% | 80.00% 이상 |
| dev precision | 99.47% | 99.00% 이상 |
| historical test recall | 71.00% | 튜닝 금지, 회귀만 기록 |
| revised hard-negative | 구현 전 재고정 | 새 FP 0 |

최종 품질 주장은 구현 전에 고정한 unseen source에서도 recall 80.00%, precision 99.00%를
동시에 통과해야 한다. 기존 Kaist·KSL test와 Korean-GSD 결과는 규칙·threshold 선택에
사용하지 않는다.

## 현재 병목

dev FN 124개 중 명사 `boundary-rejected`가 64개다. 다음으로 full-POS에서 boundary로
바뀌는 용언 12개, 대명사 6개, 수사 4개가 있다. 단순 표제어 추가로는 목표에 도달하지
못하며 기본 `smart`를 임의 substring 검색으로 완화하면 `compound-substring`
hard-negative와 충돌한다.

따라서 목표 경로는 정확한 token 분석에서 표제어·품사 component가 확인될 때만 query의
왼쪽·오른쪽 내부 경계를 증명하는 선택적 nominal-component verifier다. 임의의 합성어 분해,
corpus 단어 denylist와 fixture 전용 예외는 사용하지 않는다.

[smart component 근거](2026-07-13-smart-component-evidence.md)에서 64개 case의 query 위치와
외부 분석기 교차 결과를 기록했다. 57개는 Kiwi와 Lindera가 모두 찾고 나머지 7개도 한쪽이
찾아, morphology-aware 경로의 추가 계측을 진행할 근거가 충분하다.

## P0. 회복 가능 상한을 계측한다

1. dev의 명사 boundary FN 64개를 붙여 쓴 단위·호칭, source가 선언한 복합어 component,
   생산 접미 결합, 분석 불일치로 분류한다.
2. morphology resource의 source 분석에서 query lemma/POS component와 원문 byte span이
   일치하는 case를 report에 기록한다.
3. `사용자권한 → 권한`은 positive로 전환하고 component 경계를 가로지르는 substring,
   잘못 붙여 쓴 용언 음성 fixture를 precision negative로 확장한다.
4. case 선택과 분류는 dev gold와 고정 source 분석만 사용하고 kfind 예측은 사용하지 않는다.

완료 게이트:

- 서로 중복되지 않는 dev FN 30개 이상에서 일반 규칙 또는 source component 근거가 있다.
- 기존·신규 hard-negative에서 허용 후보가 0개다.
- 30개 미만이면 제품 verifier 구현을 시작하지 않고 목표나 resource 설계를 다시 정한다.

계측 결과 embedded는 75개 case의 87개 candidate를 평가해 accept 70, reject 17을 기록했다.
고유 accept case는 61개다. full-POS 고유 accept는 65개이며 revised hard-negative의 component
candidate 5개는 두 profile에서 모두 reject했다. 기본 test·dev 검색 결과는 변하지 않아 P0
gate를 통과했다.

## P1. resource 없는 일반 규칙을 먼저 닫는다

다음 항목은 nominal-component resource와 분리해 작은 커밋으로 처리한다.

- 숫자 뒤 단위성 명사처럼 문자 class와 고정 품사 계약만으로 증명되는 붙여쓰기
- dev에서 반복 근거가 있는 일반 용언 활용과 bounded continuation
- source 정렬·학습자 오탈자·span 불일치는 제품 규칙에서 제외

목표 기여는 중복 제거한 dev TP 4~8개다. 각 규칙은 독립 fixture, POS별 recall,
hard-negative precision과 historical test 회귀를 통과해야 한다.

## P2. nominal-component resource를 측정한다

고정 MeCab snapshot의 morphology resource에서 다음 읽기 전용 projection을 만든다.

```text
NFC token surface
  -> source analysis group
  -> component POS와 token 내부 byte span
```

lookup은 기존 `smart`가 명사 경계를 거부한 candidate에만 실행한다. token의 complete
analysis path가 query lemma/POS와 정확히 같은 component span을 포함할 때만 후보를 복구한다.
단순히 query 문자열이 token 안에 있다는 이유로 복구하지 않는다.

packed Double-Array와 FST를 동일 payload로 비교하고 artifact 크기, exact lookup,
초기화, mmap RSS를 기록한다. full morphology resource를 그대로 배포하는 안과 compact
projection을 분리해 비교한다.

path 분류 결과 embedded accept 61개는 derivational continuation 23, nominal compound 22,
particle continuation 8, copular continuation 7, mixed 1이다. P1 일반 규칙 후보는
derivational 23개이며 numeric unit 후보는 없다. reject evidence 16개는 positive 14개 case에
걸쳐 nominal 경쟁 9, unknown 3, predicate 경쟁 2개 case로 분류됐다.

이 경로들은 명사 외 node와 연결 비용 경쟁을 함께 사용하므로 compact 비교 대상은 component
span 목록이 아니라 판정 동등 lattice projection으로 좁힌다. 모든 source node의 POS·context
ID·word cost, matrix와 unknown 정의를 보존하고 판정에 쓰지 않는 source 설명 metadata만
제외한다. full resource와 lookup hit·scoring checksum이 다르면 배포 후보에서 제외한다.

완료 게이트:

- P1과 중복되지 않는 dev TP 20~30개를 회복한다.
- 기존·신규 hard-negative FP가 늘지 않는다.
- artifact는 schema, source SHA-256, section digest와 component span 범위를 검증한다.
- 기준 full-POS profile 대비 throughput·p95 회귀가 각각 10% 이내이고 peak RSS는 64 MiB
  이하, 초기화 median은 200 ms 이하다.

## P3. shadow profile에서 제품 경계를 검증한다

1. benchmark에 component-aware profile을 추가하되 기본 검색 결과는 바꾸지 않는다.
2. raw boundary reject, resource lookup, component accept/reject와 provenance를 case별로 남긴다.
3. resource 누락·손상·source 불일치를 구분하고 benchmark에서는 fallback 없이 실패한다.
4. dev에서 recall·precision·성능 게이트를 통과한 뒤에만 기본 설치 profile 적용을 제안한다.

기존 embedded profile과 `--boundary any`는 변경하지 않는다. 기본 `smart`의 결과를 바꾸는
단계에서는 CLI, Rust/WASM API, explain/JSON, 배포 artifact와 resource 실패 정책을 스펙에
먼저 확정한다. 지정사 P3 filtering과 비용 threshold 변경은 이 계획에 포함하지 않는다.

## P4. 독립 검증과 완료 판정

unseen source는 결과를 보기 전에 다음을 고정한다.

- 재배포 가능한 license와 source revision·SHA-256
- lemma와 세부 POS, 원문 span 정렬 지원
- 기존 Kaist·KSL dev/test, Korean-GSD와 NFC 문장 hash 중복 0건
- deterministic 전수 선택 또는 quota·seed·fixture digest

dev에서만 규칙과 자료구조를 확정한 뒤 historical test와 unseen source를 한 번 실행한다.
unseen source가 recall 80.00%, precision 99.00%를 동시에 통과하지 못하면 case별 보정 없이
실패 결과를 regression baseline으로 전환한다.

## 작업 단위

1. FN 분류와 독립 positive/negative evidence (shadow 완료, 최저 비용 path 유형 분류 진행 중)
2. resource 없는 일반 규칙
3. nominal-component artifact와 자료구조 benchmark
4. shadow verifier와 report profile
5. 제품 적용 스펙과 구현
6. unseen 평가와 handoff 갱신

각 단위는 fmt, workspace clippy/test, morphology Docker benchmark를 통과한 독립 커밋과 PR로
나눈다. artifact나 benchmark 경로가 바뀌는 단위는 morphology-index benchmark도 실행한다.

path 유형은 case별 accept/reject 최저 비용 evidence에서 하나씩 선택한다. accept는
`numeric-unit`, `derivational-continuation`, `copular-continuation`, `nominal-compound`,
`particle-continuation`, `exact-token`, `mixed`로 분류한다. reject는 최저 경쟁 경로를
`unknown`, `nominal`, `predicate`, `mixed`로 분류하고 POS sequence와 unknown node 유무를
보존한다. P1 중복은 구현 완료가 아니라 일반 규칙 후보로만 집계한다.
